**Step A: Add row-span calculation to useGridResize.ts**

Update the pointermove handler to also compute vertical resize:
- On pointerdown, compute `rowPx`: find the card element via `gridRef.current?.querySelector(`[data-widget-id="${id}"]`)` and compute `element.offsetHeight / startH`. If not found, use a default of 200px.
- On pointermove, compute deltaY = `e.clientY - startY`. New h = `startH + Math.round(deltaY / rowPx)`. Clamp h to [1, 4] (max 4 rows -- prevents cards from being excessively tall).
- Call `onResize(id, { w: newW, h: newH })` whenever either dimension changes.

**Step B: Add grid-auto-rows and row-span to DashboardShell.tsx**

1. Set a fixed row height on the grid container using `grid-auto-rows`:
```tsx
<div ref={gridRef} className="grid grid-cols-1 md:grid-cols-6 gap-3"
  style={{ gridAutoRows: '200px' }}
>
```
The 200px base row height keeps cards consistent. Cards with h > 1 will span multiple rows.

2. In WidgetCard, add `data-widget-id={id}` attribute for the resize hook to find elements.

3. Apply row-span via inline style on WidgetCard:
```tsx
const rowSpan = dims.h > 1 ? { gridRow: `span ${dims.h}` } : undefined;

// Merge with existing style object:
const combinedStyle = { ...style, ...rowSpan };
```

4. Update DEFAULT_DIMS -- all default h values are already 1, which is correct. The heatmap and performance tables could benefit from h: 2 as defaults, but keep h: 1 for now per user decision ("include h in data model but do not expose row-height resize UI yet" -- however since this plan explicitly implements RESIZE-02, we ARE exposing it now, so use sensible defaults):
```typescript
"heatmap": { w: 1, h: 2 },       // calendar needs vertical space
"risk-simulator": { w: 3, h: 2 }, // simulator needs vertical space
```

**Step C: Debounce Recharts ResponsiveContainer during resize**

Per STATE.md decision: "Debounce Recharts ResponsiveContainer during resize to prevent performance collapse."

1. In DashboardShell, create a `isResizing` boolean derived from `resizingId !== null`.
2. Pass `isResizing` down to WidgetCard. When `isResizing && id === resizingId`, render a placeholder instead of chart content:
```tsx
{isBeingResized ? (
  <div className="flex-1 flex items-center justify-center dark:text-slate-500 text-slate-400 text-sm">
    {dims.w} x {dims.h}
  </div>
) : children}
```
This shows the current dimensions during resize (like Grafana's resize preview) and prevents Recharts from thrashing. When resize ends (pointerup), the actual chart renders at the new size once.

3. Add `select-none` class to the grid container when resizing to prevent text selection during drag.

**Step D: Handle @dnd-kit interaction with variable row spans**

Per STATE.md concern: "@dnd-kit rectSortingStrategy has known issues with variable-sized items."

If row spans > 1 cause dnd-kit sorting glitches, switch from `rectSortingStrategy` to `null` (no strategy):
```tsx
<SortableContext items={visibleWidgets} strategy={null}>
```
This disables sorting animation hints but keeps drag-and-drop functional. Test both approaches -- if rectSortingStrategy works acceptably, keep it.

**Step E: Backward compatibility for row-span in old layouts**

The migration in Plan 01 already sets `h: 1` for old layouts. No additional migration needed. But ensure that if a loaded dims entry somehow lacks `h`, it defaults to 1:
```typescript
const d = layout.dims[id] ?? DEFAULT_DIMS[id] ?? { w: 1, h: 1 };
const dims = { w: d.w ?? 1, h: d.h ?? 1 };
```
